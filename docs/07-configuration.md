# Configuration Reference

Complete schema for `openspec/config.yaml` — the SDD configuration file generated by `sdd init`.

---

## Overview

`openspec/config.yaml` is the single source of truth for SDD configuration in a project. It captures:
- Technology stack (auto-detected by `sdd init`)
- Architecture patterns (monorepo vs single-package)
- Project conventions (extracted from `CLAUDE.md`)
- Commands for build, lint, and test (consumed by `sdd verify`)
- Phase-specific rules for each SDD agent

The file is generated by `sdd init` (Go code, zero tokens) and read by every subsequent SDD phase — both the Go binary and the sub-agents that run inside the LLM.

---

## Auto-detection: `sdd init`

`sdd init` runs entirely in Go. It does not invoke the LLM. It walks the project directory, reads known config files, and writes `openspec/config.yaml`.

```
sdd init
# → reads package.json / go.mod / pyproject.toml / Cargo.toml
# → reads tsconfig.json for language version
# → reads CLAUDE.md for conventions
# → detects lockfile to determine runtime
# → detects frameworks from dependency names
# → writes openspec/config.yaml
# → writes openspec/AGENTS.md (template if not present)
# Zero LLM calls. Completes in < 1 second.
```

Stack detection covers 6 language ecosystems:

| Lockfile / manifest | Detected runtime | Language |
|---|---|---|
| `bun.lockb` | `bun` | TypeScript / JavaScript |
| `yarn.lock` | `node` | TypeScript / JavaScript |
| `pnpm-lock.yaml` | `node` | TypeScript / JavaScript |
| `package-lock.json` | `node` | TypeScript / JavaScript |
| `go.mod` | `go` | Go |
| `pyproject.toml` | `python` | Python |
| `Cargo.toml` | `rust` | Rust |

Framework detection (within a detected runtime) reads dependency names from the manifest to populate `stack.frameworks.*`.

Re-run `sdd init` at any time to update the config. Use `sdd init --force` to overwrite including the `contracts` section.

## Full Schema

```yaml
# openspec/config.yaml
# Generated by sdd init. Edit with care — the sdd binary and all sub-agents read this file.

schema: spec-driven
version: "1.0"
generated_at: "2026-02-22T14:30:00Z"   # ISO 8601 timestamp

# ─────────────────────────────────────────────────────────
# PROJECT
# ─────────────────────────────────────────────────────────
project:
  name: my-project                # From package.json "name" or directory name
  path: /home/user/my-project     # Absolute path to project root
  type: monorepo                  # monorepo | single-package

# ─────────────────────────────────────────────────────────
# STACK
# Auto-detected by sdd init from package.json, lockfiles, tsconfig.json, etc.
# ─────────────────────────────────────────────────────────
stack:
  runtime: bun                    # bun | node | deno | go | python | rust
  language: typescript            # typescript | javascript | go | python | rust
  language_version: "ES2022"      # From tsconfig.json "target"
  frameworks:
    frontend: react               # react | vue | svelte | none
    backend: elysia               # elysia | express | fastify | hono | none
    testing: bun:test             # bun:test | vitest | jest | pytest | go test | cargo test | none
    meta_framework: nextjs        # nextjs | remix | nuxt | none
  database: postgresql            # postgresql | mysql | sqlite | none
  orm: drizzle                    # drizzle | prisma | typeorm | none
  state_management: zustand       # zustand | jotai | redux | none
  css_framework: tailwind         # tailwind | none

# ─────────────────────────────────────────────────────────
# ARCHITECTURE
# ─────────────────────────────────────────────────────────
architecture:
  pattern: monorepo               # monorepo | single-package
  workspaces:                     # List of workspace paths (if monorepo)
    - apps/web
    - apps/api
    - packages/ui
    - packages/shared
  entry_points:
    frontend: apps/web/src/app/page.tsx
    backend: apps/api/src/index.ts

# ─────────────────────────────────────────────────────────
# SKILLS
# Controls which SKILL.md files the Go assembler includes in context.
# ─────────────────────────────────────────────────────────
skills:
  path: ~/.claude/skills          # Root directory for all SKILL.md files
  # The assembler automatically includes:
  #   skills/sdd/{phase}/SKILL.md          (current phase)
  #   skills/frameworks/{f}/SKILL.md       (each matched framework from stack.*)
  #   skills/analysis/{skill}/SKILL.md     (when parent phase includes it)
  # Additional skills to always include (project-specific):
  extra: []
  # Skills to exclude even if stack-matched:
  exclude: []

# ─────────────────────────────────────────────────────────
# COMMANDS
# Used by `sdd verify` (Go) and exposed to sub-agents via {CMD_*} variables.
# `sdd init` populates these from the detected stack automatically.
# ─────────────────────────────────────────────────────────
commands:
  build: bun run build            # Full build (may be slow — skipped if smart_skip detects no changes)
  typecheck: bun run typecheck    # Fast type-check without emit
  lint: bun run lint              # Lint check
  lint_fix: bun run lint:fix      # Lint with auto-fix
  format_check: bun run format:check
  format_fix: bun run prettier --write
  test: bun test                  # Run all tests
  test_watch: bun test --watch    # Watch mode (used by sdd apply internally)

# Per-language defaults filled by sdd init:
# Go:     build=go build ./...  typecheck=go vet ./...  lint=golangci-lint run  test=go test ./...
# Python: build=python -m build  typecheck=mypy .  lint=ruff check .  test=pytest
# Rust:   build=cargo build  typecheck=cargo check  lint=cargo clippy  test=cargo test

# ─────────────────────────────────────────────────────────
# CONVENTIONS
# Extracted from CLAUDE.md by sdd init. Used by all sub-agents.
# ─────────────────────────────────────────────────────────
conventions:

  type_strictness:
    # Patterns banned in ALL production code
    # Examples (TypeScript): any, "as Type", @ts-ignore, @ts-expect-error, non-null assertion
    # Examples (Python): type: ignore, Any without constraint
    # Examples (Go): interface{} instead of any (Go 1.18+), unchecked error returns
    banned: []

    # Patterns allowed everywhere
    allowed: []

    # Patterns allowed ONLY in test files
    test_only: []

  error_handling:
    pattern: result               # result | throw | either — detected from project conventions
    # result_type_path: optional, set if project has a central error type definition
    empty_catch: forbidden        # empty | log | forbidden

  testing:
    runner: ""                    # e.g., bun:test, vitest, jest, pytest, go test, cargo test
    pattern: describe-it          # describe-it | describe-test | function-based
    file_suffix: ""               # e.g., .test.ts, .spec.ts, _test.go, _test.py
    location: colocated           # colocated | __tests__ | tests/
    # Example: feature.ts → feature.test.ts in same directory
    one_assertion_per_it: true    # Aim for one assertion per it() block
    preferred_mocking: dependency-injection  # DI preferred over mocking
    mock_function: mock           # From bun:test: mock() for function mocks

  file_organization:
    max_lines_warning: 600        # Files over this → warning
    max_lines_error: 800          # Files over this → error, must split
    max_nesting_depth: 3          # Max levels of nesting (use early returns)
    single_responsibility: true   # One responsibility per file
    test_location: colocated      # Test files next to source files

  code_style:
    async_preference: async-await # async-await | then-chains
    immutability: prefer-immutable  # prefer-immutable | mutable
    local_mutations_ok: true      # push/pop on LOCAL arrays is fine
    no_console_log: true          # Use structured logger instead
    no_magic_numbers: true        # Use named constants
    no_magic_strings: true        # Use named constants

  security:
    no_eval: true
    no_new_function: true
    no_innerHTML_dynamic: true    # Dynamic innerHTML is XSS risk
    validate_at_boundaries: true  # Validate all user input at API boundaries
    no_commit_secrets: true       # .env, *.pem, *.key, credentials.json

  git:
    no_co_authored_by: true       # No "Co-Authored-By: Claude" in commits
    no_ai_attribution: true       # No Claude attribution in commit messages

# ─────────────────────────────────────────────────────────
# CODE REVIEW (AGENTS.md)
# ─────────────────────────────────────────────────────────
code_review:
  agents_md: AGENTS.md            # Path to AGENTS.md relative to project root
  # AGENTS.md uses keyword-prefixed rules:
  # REJECT: violations that must block merge
  # REQUIRE: conditions that must be met
  # PREFER: advisory best practices

# ─────────────────────────────────────────────────────────
# PHASES
# Rules for each SDD pipeline phase
# ─────────────────────────────────────────────────────────
phases:

  proposal:
    required_sections:
      - intent
      - scope
      - approach
      - affected_areas
      - risks
      - rollback_plan
      - dependencies
      - success_criteria
    large_change_threshold: 10    # Files touched — warn to split if exceeded

  specs:
    keywords: RFC2119             # MUST/SHALL/SHOULD/MAY keyword standard
    scenario_format: given-when-then
    min_scenarios_per_requirement: 1  # Every MUST needs at least one scenario
    concrete_values: required     # Scenarios must use specific values, not placeholders

  design:
    required_sections:
      - technical_approach
      - architecture_decisions
      - data_flow
      - file_changes
      - interfaces
      - testing_strategy
    alternatives_per_decision: 2  # Every decision needs 2+ alternatives considered

  tasks:
    ordering: bottom-up          # Phases determined dynamically by dependency analysis
    ordering_hint: ""            # Optional: domain-specific hint (e.g., "database before API")
    task_format: "N.M Action — file, change"  # e.g., "2.1 Create — /abs/path, implement X"
    max_files_per_task: 2        # Tasks should modify 1-2 closely related files

  apply:
    batch_size: 1                # One phase at a time
    verify_after_each: true      # Run build-fix loop after each batch
    max_fix_attempts: 5          # Per unique error before flagging for manual review

  review:
    checklist:
      - spec_compliance
      - design_compliance
      - pattern_compliance
      - agents_md_rules
      - naming_readability
      - security_scan

  verify:
    # `sdd verify` reads these from `commands:` above, not from this section.
    # This section controls quality gate thresholds.
    fail_on_warnings: false       # Warnings = PASS WITH WARNINGS, not FAIL
    spec_test_coverage_fail: 50  # FAIL if >50% of spec scenarios have no tests
    smart_skip:
      enabled: true              # Skip expensive steps when inputs haven't changed
      # sdd verify skips `commands.build` if no source files changed since last green run.
      # Always runs typecheck + lint + test regardless of smart_skip.

  clean:
    scope: change_plus_dependents # Only clean files from current change + direct deps
    rule_of_three: true          # Consolidate only if pattern appears 3+ times
    verify_after_removal: true   # Run typecheck+tests after CAREFUL removals

  archive:
    date_format: "YYYY-MM-DD"
    preserve_removed_specs: true  # Comment out, never delete REMOVED requirements
    learnings_threshold: reusable # Only save genuinely reusable, non-obvious insights

# ─────────────────────────────────────────────────────────
# CONTRACTS (v1.1)
# Pre/post-conditions for each phase, validated by the Go orchestrator
# ─────────────────────────────────────────────────────────
contracts:
  explore:
    preconditions:
      - config.yaml exists at openspec/config.yaml
      - topic is non-empty string
    postconditions:
      - exploration.md written to openspec/changes/{changeName}/ (if changeName provided)
      - envelope contains relevant_files array with ≥1 entry
      - envelope status is success or error
  propose:
    preconditions:
      - exploration.md exists at openspec/changes/{changeName}/
    postconditions:
      - proposal.md written with all required_sections
      - envelope contains approach and risk_summary
  spec:
    preconditions:
      - proposal.md exists and was approved
    postconditions:
      - ≥1 spec file in openspec/changes/{changeName}/specs/
      - each spec contains ≥1 GIVEN/WHEN/THEN scenario
  design:
    preconditions:
      - proposal.md exists and was approved
    postconditions:
      - design.md written with all required_sections
      - ≥1 interface definition with typed fields
  tasks:
    preconditions:
      - spec files exist in specs/
      - design.md exists
    postconditions:
      - tasks.md written with ≥1 phase containing ≥1 task
      - each task references a spec scenario or design component
  apply:
    preconditions:
      - tasks.md exists with ≥1 uncompleted task
      - design.md exists
      - spec files exist in specs/
    postconditions:
      - ≥1 task marked [x] in tasks.md
      - buildStatus included in envelope
      - all created/modified files listed
  review:
    preconditions:
      - ≥1 task marked [x] in tasks.md
      - files created/modified by apply exist on disk
    postconditions:
      - review-report.md written
      - envelope contains verdict (PASSED or FAILED)
      - all spec scenarios accounted for in coverage matrix
  verify:
    preconditions:
      - review-report.md exists (or explicitly skipped)
      - implementation files exist on disk
    postconditions:
      - verify-report.md written
      - envelope contains buildHealth with all 4 checks
      - verdict consistent with decision matrix
  clean:
    preconditions:
      - verify verdict is PASS or PASS_WITH_WARNINGS
    postconditions:
      - specs merged to openspec/specs/ (if enabled)
      - no orphaned imports or dead code in changed files
  archive:
    preconditions:
      - clean phase completed
    postconditions:
      - change directory moved to archive/
      - summary.md written with final metrics
```

## Field Reference

### `schema` and `version`

```yaml
schema: spec-driven   # Always "spec-driven"
version: "1.0"        # SDD schema version
```

Used by the `sdd` binary and sub-agents to verify they are reading a valid SDD config.

---

### `stack.runtime`

Detected by `sdd init` by checking lockfiles in this order:
1. `bun.lockb` → `bun`
2. `yarn.lock` → `node` (yarn)
3. `pnpm-lock.yaml` → `node` (pnpm)
4. `package-lock.json` → `node` (npm)
5. `go.mod` → `go`
6. `pyproject.toml` → `python`
7. `Cargo.toml` → `rust`

---

### `commands`

The `commands` block is the authoritative source for how to build, lint, and test the project. `sdd verify` reads this block directly — it does not ask the LLM what commands to run.

```yaml
commands:
  build: go build ./...
  typecheck: go vet ./...
  lint: golangci-lint run
  test: go test ./...
```

Sub-agents reference these via template variables: `{CMD_BUILD}`, `{CMD_TYPECHECK}`, `{CMD_LINT}`, `{CMD_TEST}`. The Go assembler resolves these before sending context to the LLM.

---

### `skills.path`

Default: `~/.claude/skills`. Override if your team stores skills in a different location:

```yaml
skills:
  path: /shared/team-skills
```

The Go assembler resolves `~` to the current user's home directory.

---

### `conventions.error_handling.result_type_path`

The path to your `Result<T, E>` type definition. `sdd-apply` reads this file before implementing to understand the exact type signature.

```typescript
// Example: src/shared/result.ts
export type Result<T, E> = Ok<T> | Err<E>
export type Ok<T> = { readonly ok: true; readonly value: T }
export type Err<E> = { readonly ok: false; readonly error: E }
```

If your project doesn't use Result, set `pattern: throw` and remove `result_type_path`.

---

### `code_review.agents_md`

Path to `AGENTS.md` relative to project root. Example AGENTS.md:

```markdown
# AGENTS.md — AI Code Review Rules

REJECT: Direct database queries outside src/data/ — all DB access through repository layer
REJECT: console.log in production code — use the structured logger from src/logger.ts
REJECT: any type in production code — use unknown + type guards

REQUIRE: All API routes must validate request body with Zod before processing
REQUIRE: All new public functions must have explicit TypeScript return types
REQUIRE: Error handling must use Result<T, E> for fallible operations

PREFER: Branded types for domain IDs (UserId, OrderId) over plain strings
PREFER: Early returns to reduce nesting depth
PREFER: Named constants over magic numbers
```

---

### `phases.apply.max_fix_attempts`

How many times `sdd-apply` attempts to fix a single build error before escalating to manual review. Default: 5.

Set lower (3) for simpler projects where errors should be clear. Set higher (7) for very complex TypeScript generics.

---

### `phases.verify.spec_test_coverage_fail`

Percentage threshold for failing verification when spec scenarios lack test coverage.

```yaml
spec_test_coverage_fail: 50  # FAIL if >50% of scenarios have no tests
```

Set to `0` to require 100% scenario coverage. Set to `100` to never fail on coverage (useful early in development).

---

### `phases.verify.smart_skip`

When `smart_skip.enabled: true`, `sdd verify` skips the `commands.build` step if no source files have changed since the last green verify run. Content hashes of source files are stored in `openspec/.verify-cache`. TypeCheck, lint, and test always run.

```yaml
phases:
  verify:
    smart_skip:
      enabled: true   # default: true
```

---

### `contracts`

*Added in v1.1.* Formal pre/post-conditions for each SDD phase.

**Preconditions** are validated by the Go orchestrator BEFORE launching a sub-agent. If any precondition fails, the sub-agent is not launched and the orchestrator reports which prior phase needs to run first.

**Postconditions** are validated AFTER the sub-agent returns. Failures are logged as warnings in `quality-timeline.jsonl` but do not block the next phase.

If the `contracts` section is missing (projects initialized before v1.1), contract validation is skipped. Run `sdd init --force` to regenerate with contracts.

## Minimal Config

For projects that don't use TypeScript or have simpler conventions:

```yaml
schema: spec-driven
version: "1.0"
generated_at: "2026-02-22T00:00:00Z"

project:
  name: my-project
  path: /home/user/my-project
  type: single-package

stack:
  runtime: node
  language: javascript
  frameworks:
    testing: jest

commands:
  typecheck: echo "No TypeScript"
  lint: npm run lint
  test: npm test

conventions:
  error_handling:
    pattern: throw
  testing:
    runner: jest
    pattern: describe-it
    file_suffix: .test.js
    location: colocated
  file_organization:
    max_lines_warning: 600
    max_lines_error: 800

phases:
  verify:
    fail_on_warnings: false
    spec_test_coverage_fail: 50
```

Note: the `commands` block is now a top-level key (not nested under `phases.verify`). `sdd init` generates this layout automatically for all detected stacks.

## Updating config.yaml

`sdd init` generates the config but you can edit it manually. Common edits:

**Adjust file size limits:**
```yaml
conventions:
  file_organization:
    max_lines_warning: 400  # Stricter than default 600
```

**Add a custom AGENTS.md location:**
```yaml
code_review:
  agents_md: .github/AGENTS.md  # Non-standard location
```

**Add additional verify commands:**
```yaml
commands:
  typecheck: bun run typecheck
  lint: bun run lint
  test: bun test
  e2e: bun run playwright test   # sdd verify runs this too
```

**Override skill path for team-shared skills:**
```yaml
skills:
  path: /shared/company-skills
  extra:
    - internal/our-api-conventions  # Always include this skill
```
